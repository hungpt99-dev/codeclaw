# Step 50: Git Diff Review Enhancement in Web UI

## Status

Planned

## Priority

P3

## Product Goal

Enhance the web UI diff viewer to provide a visual, line-by-line code review experience. Users can review changed files side-by-side, see diff statistics, and approve or reject changes directly from the UI. This transforms the web UI from a passive artifact viewer into an active code review tool.

## Problem

The current web UI has a basic `DiffViewer` component (`apps/local-web/src/components/DiffViewer.tsx`) that displays raw diff text. There is no:
- Side-by-side or unified diff view toggle
- Syntax highlighting in diff
- Per-file diff navigation
- Diff statistics (additions, deletions, files changed)
- File-level approve/skip actions
- Line-level commenting
- Integration with approval gates

This makes it difficult for users to review code changes generated by AI CLIs within the web UI.

## Current Evidence

- `apps/local-web/src/components/DiffViewer.tsx` — basic diff text display
- `packages/adapters/src/diff/diffService.ts` — generates diff patches
- `packages/adapters/src/git/gitService.ts` — collects changed files and diff stats
- No syntax highlighting in diff display
- No side-by-side view
- No diff statistics display in UI
- No file-level navigation for multi-file diffs
- No approve/reject actions for individual files

## Scope

### In Scope

- Enhanced DiffViewer with syntax highlighting (using highlight.js or similar)
- Side-by-side and unified view toggle
- File-level navigation for multi-file diffs
- Diff statistics: files changed, additions, deletions per file
- File status indicators (added, modified, deleted, renamed)
- Per-file approve/skip buttons for review workflow
- Integration with approval gates for risky files
- Diff tab in run detail showing all changed files

### Out of Scope

- Line-level commenting
- Inline code editing
- GitHub-style pull request review
- Image diff comparison

## Expected User Value

Users can review AI-generated code changes directly in the web UI with proper syntax highlighting, file navigation, and statistics. The diff review becomes a meaningful checkpoint before accepting changes, not just a raw text display.

## Expected Behavior

1. After code execution, "Code Changes" tab appears in run detail
2. Tab shows diff statistics: X files changed, Y additions, Z deletions
3. File list with status indicators (added/modified/deleted)
4. Selecting a file shows its diff with syntax highlighting
5. User can toggle between unified and side-by-side views
6. User can mark individual files as reviewed or flag them
7. Risky file modifications (matching warnFiles patterns) are highlighted
8. Diff view integrates with approval gates

## Security Considerations

Diff content may include code that contains secrets. The diff viewer must respect the same safety rules as the code execution — protected file modifications should be flagged.

## Risks

- Syntax highlighting library may increase bundle size
- Side-by-side view on small screens is challenging
- Large diffs may cause performance issues
- Diff parsing for syntax highlighting may not match all languages

## Dependencies

- Requires code execution output (diff patches, changed files)
- Builds on existing `diffService.ts` and `gitService.ts`
- Web UI must have basic diff component (already exists)

## Notes for AI Coding Agent

Use a lightweight syntax highlighting library. Consider `diff2html` for side-by-side HTML diff rendering. For the file list, use virtual scrolling if there are many files. Keep the component modular so it can be reused in approval screens.
